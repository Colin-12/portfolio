Contexte & Objectif

Besoin marché identifié : les recruteurs (Arkéa, ESN) cherchent des AI Engineers capables de construire des systèmes RAG de production — pas juste des chatbots. Ce projet démontre : architecture, sécurité, gouvernance, pilotage des coûts, UI/UX.

Simuler l'outil d'un analyste dans un département financier bancaire qui interroge des rapports annuels (résultats Arkéa 2025, URD 2024, SFH 2025) et de la documentation réglementaire (ACPR 2024). Le système doit répondre avec des chiffres exacts, des références normatives, et bloquer les questions hors périmètre.

Architecture multi-agents

Pipeline complet en 6 briques : ingestion PDF → chunking sémantique → indexation → guards → routing → agents spécialisés.

1
Ingestion PDF → Markdown structuré
pymupdf4llm convertit chaque PDF en Markdown. Les tableaux financiers (PNB, CET1, résultats) deviennent du texte structuré avec leurs valeurs préservées — là où PyMuPDF brut les éclatait.
pymupdf4llm4 PDFsTableaux préservés
2
Chunking sémantique en 2 passes
Passe 1 : MarkdownHeaderTextSplitter découpe par sections (#, ##, ###). Passe 2 : RecursiveCharacterTextSplitter sous-découpe les sections longues. Chaque chunk conserve son chemin de section en métadonnée.
MarkdownHeaderTextSplitter1000 tokens3 890 chunks
3
Indexation + Hybrid Search
2 collections ChromaDB (rapports / réglementation). Recherche hybride : embeddings sémantiques (all-MiniLM-L6-v2) + BM25 lexical + fusion Reciprocal Rank Fusion. Le BM25 trouve les termes exacts (CET1, PNB), les embeddings capturent le sens.
ChromaDBBM25RRFHuggingFace
4
Guards de sécurité (2 niveaux)
Niveau 1 (statique) : regex pour injections et hors-sujet évident (< 1ms). Niveau 2 (LLM) : le modèle rapide 8B évalue la pertinence métier (~90ms). Les questions hors périmètre sont rejetées avant d'atteindre les agents.
RegexLLM 8BFail-open
5
Router LLM → 3 agents spécialisés
Le router (Llama 8B) classifie chaque question vers l'agent approprié. Chaque agent a son propre ChatPromptTemplate, sa logique de retrieval, et son format de réponse adapté au domaine.
AgentRapportsAgentRéglementationAgentDirect
6
Orchestrateur central
Classe Orchestrateur qui gère le cycle complet : guards → routing → sélection agent → exécution → log JSON. Chaque requête produit une trace complète (latences, sources, confiance) pour le monitoring.
LoggingJSONMonitoring

Itération V1 → V2

Diagnostic des causes racines puis refonte complète — pas de patches sur une base fragile.

V1 — Fragile
PyMuPDF brut — tableaux financiers éclatés
RecursiveCharacterTextSplitter aveugle
Pipeline linéaire (query → retrieve → LLM)
Routing vers des collections, pas des agents
Patches empilés V1→V2→V3→V4
V2 — Solide
pymupdf4llm → Markdown structuré
MarkdownHeaderTextSplitter sémantique
3 agents spécialisés avec prompts dédiés
Routing vers des agents à comportements distincts
Architecture propre dès le départ
Décision clé : recommencer sur des bases solides (3-4h) plutôt que patcher l'existant (1h). Le résultat : un système présentable en production, pas un prototype déguisé.
