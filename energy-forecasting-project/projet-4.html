Forecasting & Détection de Pics Énergétiques

Proof of Concept sur données réelles : ce projet s’appuie sur le dataset public ASHRAE Great Energy Predictor III. Le pipeline a été développé sur les mesures électriques (meter = 0) et illustré sur un bâtiment unique afin de valider proprement la démarche de prévision et de détection de pics avant généralisation multi-bâtiments.

1. Contexte Métier & Objectif

Les infrastructures techniques fortement consommatrices d’électricité doivent anticiper les hausses de charge afin de mieux piloter l’exploitation, limiter les coûts énergétiques et réduire les risques opérationnels. L’objectif de ce projet est double : prévoir la consommation horaire et détecter les épisodes de forte demande.

2. Données & Périmètre

  • Source : ASHRAE Great Energy Predictor III.
  • Chargement initial : 2 000 000 lignes du fichier train.csv dans Colab.
  • Filtrage : conservation des compteurs électriques uniquement (meter = 0).
  • Fusion : consommation + météo + métadonnées bâtiments.
  • Cas d’étude modélisé : un bâtiment individuel pour valider proprement le pipeline de forecasting.
  • Split : Train / Validation / Test strictement chronologique pour éviter le data leakage.

3. Préparation des Données & Feature Engineering

Dans un problème de séries temporelles, la qualité du modèle dépend fortement de la manière dont on représente la dynamique du passé. J’ai construit un pipeline de variables orienté inertie et saisonnalité :

  • Lags : H-1, H-2, H-24, H-48, H-72, H-168.
  • Rolling features : moyennes, écarts-types, minimums et maximums glissants.
  • Calendrier : heure, jour, semaine, week-end.
  • Météo : température, humidité, vent et leurs retards.
  • Encodage cyclique : représentation sin/cos pour l’heure et le jour de semaine.

Point méthodologique clé :

Le split Train / Validation / Test a été réalisé de manière strictement chronologique. Cette précaution évite qu’un modèle de séries temporelles n’apprenne indirectement à partir du futur.

4. Modélisation

J’ai adopté une approche en deux volets afin de répondre à deux usages métiers complémentaires :

A. Forecasting continu (Régression)

Objectif : prédire la consommation exacte à l’heure suivante. Trois approches ont été comparées : Baseline naïve, Régression Linéaire et Random Forest Regressor.

Modèle Jeu MAE RMSE MAPE
Baseline naïve Test 7.94 10.02 7.82%
Régression Linéaire Test 7.72 9.33 7.59%
Random Forest Test 5.27 6.69 5.04%

Le modèle Random Forest réduit nettement l’erreur par rapport à la baseline naïve, ce qui montre qu’un modèle non linéaire capte mieux les interactions entre historique de consommation, calendrier et conditions météo.

Fig. 1 — Le modèle Random Forest suit correctement le profil global de consommation et améliore la baseline naïve, même s’il tend à lisser légèrement certains pics extrêmes.

B. Détection de pics (Classification)

Objectif : détecter les épisodes de forte consommation au-dessus d’un seuil défini au 85e percentile. Cette formulation transforme le problème en système d’alerte énergétique.

Modèle Jeu Accuracy Precision Recall F1-score
Naive Peak Baseline Test 0.896 0.333 0.375 0.353
Logistic Regression Test 0.755 0.219 0.875 0.350
Random Forest Classifier Test 0.840 0.304 0.875 0.452

Le classifieur Random Forest atteint un recall de 87.5% sur le jeu de test, ce qui signifie qu’il identifie la grande majorité des pics réels. Dans une logique d’alerte, ce critère est particulièrement important : rater un pic coûte souvent plus cher que générer quelques fausses alertes.

Fig. 2 — Les pics prédits suivent globalement les épisodes de forte consommation. Le modèle privilégie le rappel, au prix de quelques fausses alertes acceptables en contexte opérationnel.

5. Explicabilité & Insights Opérationnels

Fig. 3 — Les variables les plus importantes sont lag_1, lag_24 et hour, ce qui confirme une forte inertie court terme et un cycle journalier marqué.

  • Inertie forte : la consommation récente est le principal signal prédictif.
  • Cycle journalier : le comportement de la veille à la même heure reste très informatif.
  • Météo secondaire : sur ce cas d’étude, les variables météo pèsent moins que la dynamique interne du bâtiment.
  • Usage opérationnel : ce type de modèle peut alimenter un système d’alerte pour décaler certaines charges non critiques et limiter les épisodes de tension énergétique.

6. Limites & Perspectives

Limites actuelles :

  • La démonstration de modélisation est réalisée à l’échelle d’un bâtiment unique.
  • Le pipeline doit encore être généralisé en version multi-bâtiments.
  • La détection des pics dépend du seuil retenu (ici 85e percentile).
  • Les résultats sont solides pour un proof of concept, mais ne constituent pas encore un système de production industriel.

Prochaines étapes envisagées : extension multi-bâtiments, validation croisée temporelle plus robuste, comparaison avec XGBoost/LightGBM, et intégration dans un dashboard de monitoring.
