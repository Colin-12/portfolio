Business Intelligence : Modélisation en Étoile & Gouvernance

Note : Ce projet utilise le dataset public "Superstore Sales". L'objectif est de démontrer une démarche BI complète : de la structuration d'un modèle de données gouverné jusqu'à la restitution via un Dashboard interactif garantissant la justesse des métriques.
Source : Superstore Sales Dataset (Raw CSV)

1. Contexte & Enjeux

Dans de nombreuses entreprises retail, le reporting repose sur des fichiers d'extraction plats. Cela entraîne des lenteurs, des erreurs de calcul (surcomptages), et l'absence d'une "Single Source of Truth" (Vérité Unique). Ce projet illustre la transition d'un fichier plat vers un pipeline analytique robuste et audité.

2. Modélisation (Star Schema) & Grain

La première étape a consisté à normaliser la donnée brute vers un Modèle en Étoile.

  • Définition du Grain : La table de faits (fact_sales) est définie à la granularité la plus fine : 1 ligne = 1 produit au sein d'une commande spécifique, à une date donnée.
  • Dimensions créées : dim_customer, dim_product, dim_location (avec création d'une Surrogate Key), et l'indispensable dim_date pour sécuriser les analyses temporelles.

3. Data Quality (DQ) & Audit post-ETL

La confiance dans la donnée est le socle de la BI. Le pipeline Python intègre des assertions strictes (tests unitaires) validant la jointure :

  • Test de non-déperdition : Somme(CA Brut) == Somme(CA Table de Faits). Écart mesuré : 0.00$.
  • Intégrité Référentielle : 0 clé étrangère orpheline ou manquante détectée.
  • Validation Métier : 100% des remises (discounts) sont comprises dans l'intervalle [0, 1].

4. Gouvernance des Métriques & Dashboarding D3.js

Le piège classique en BI est d'agréger les données en amont (ex: par mois), puis de sommer des "Count Distinct" (comme le nombre de clients ou de commandes) dans le dashboard, créant un surcomptage massif.

Ma solution : L'export JSON contient la table jointe à sa granularité fine. C'est l'interface D3.js qui recompose à la volée les métriques additives (Ventes, Profit) par simple somme, et les métriques semi-additives/non-additives (Nb Commandes) via l'utilisation stricte de Set() pour garantir des dédoublonnages parfaits selon les filtres choisis par l'utilisateur.

5. Insights Actionnables extraits du Dashboard

Une fois la donnée fiabilisée, l'outil révèle de vraies dynamiques métier :

  • Volume vs Rentabilité : La catégorie "Furniture" génère un fort Chiffre d'Affaires mais détruit la rentabilité globale (taux de marge très faible par rapport aux fournitures de bureau).
  • Forte saisonnalité Q4 : Le mois de Novembre concentre systématiquement le pic d'activité, nécessitant une anticipation logistique majeure.

6. Limites & Hypothèses assumées

  • Proxy Promotionnel : Le dataset public ne contenant pas de mécanique fidélité explicite, la règle "Toute ligne avec Discount > 0 est une Opération Promotionnelle" a été assumée comme proxy analytique.
  • SCD Type 1 : Pour les dimensions Clients et Produits, le pipeline conserve uniquement la dernière occurrence connue (Slowly Changing Dimension Type 1) pour simplifier le modèle sur ce volume restreint.
