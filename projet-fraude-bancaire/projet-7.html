Contexte & objectif

Problématique : Un département Lutte contre la Fraude Externe (LCFE) de banque de détail pilote son dispositif via des règles manuelles configurées dans un outil de détection. Ces règles plafonnent à un F1-score de 0,60 et génèrent des milliers de fausses alertes — un coût opérationnel direct pour les analystes. L'enjeu : dépasser les règles métier sur tous les indicateurs simultanément, tout en restant explicable pour le régulateur (ACPR) et les équipes métier.

Ce projet reproduit le cycle complet d'un Data Analyst fraude : définition des règles métier, entraînement et sélection du modèle ML, explicabilité SHAP transaction par transaction, et restitution via un dashboard de pilotage orienté Direction.

Construction du projet

1
Dataset & Analyse Exploratoire
Chargement du dataset ULB (284 807 transactions réelles anonymisées par PCA). Quantification du déséquilibre extrême : 0,17% de fraudes. Analyse temporelle et comportementale — le montant médian des fraudes est de 22€ (card testing).
Kaggle API Google Colab EDA Classe déséquilibrée
2
Règles métier style LCFE
Définition de 5 règles manuelles (montant, horaire, V14, V10, combinée) et évaluation sur précision, rappel et F1. La règle combinée atteint un rappel de 0,85 mais génère 10 847 faux positifs — intenable pour une équipe d'analystes.
Règles seuils Précision / Rappel Matrice de confusion Coût opérationnel
3
Modèles ML & Optimisation du seuil
Entraînement de 3 modèles (Random Forest, XGBoost, LightGBM) avec SMOTE pour le déséquilibre. Test de 45 seuils de décision (0,05 → 0,97) pour chaque modèle. LightGBM à seuil 0,91 remporte tous les critères : F1 0,85, précision 0,863, seulement 13 faux positifs.
SMOTE Random Forest XGBoost LightGBM Optimisation seuil AUC-ROC 0,977
4
Explicabilité SHAP — IA Act ready
Calcul des SHAP values sur 56 962 transactions. Explication transaction par transaction (waterfall plot) pour 3 cas types : fraude détectée, fraude manquée, fausse alerte. Analyse du profil des faux positifs et proposition d'une règle d'exclusion ciblée (V14 modéré + montant faible).
SHAP TreeExplainer Waterfall plot XAI / IA explicable Conformité ACPR
5
Dashboard de pilotage D3.js
Dashboard interactif orienté Direction : 4 KPIs clés, comparatif règles vs ML, courbe de seuils, importance SHAP globale, distribution des scores, matrice de confusion et analyse des 3 cas SHAP. Hébergé statiquement sur GitHub Pages.

Insight principal : L'optimisation du seuil de décision (0,5 → 0,91) améliore le F1 de LightGBM de +29% (0,66 → 0,85). Le seuil par défaut est rarement optimal sur des données bancaires déséquilibrées — cette calibration est une décision métier, pas technique.
Analyse des faux positifs : Les 13 fausses alertes présentent des valeurs V14 modérément négatives (-3,96 en moyenne vs -8,24 pour les vraies fraudes) et de faibles montants (médiane 1€). Piste d'amélioration identifiée : règle d'exclusion hybride ML+métier sur V14 ∈ [-5, -3] et Amount < 10€.
Analyse XAI — fraude manquée : La fraude non détectée (proba 13,7%) présente un signal V14 fort (+8,2 SHAP) mais est contrebalancée par 5 variables (V26, Time, V27...). Ce cas justifie une revue du paramétrage des seuils en comité opérationnel.
